1. Overview & problem

Agents working in a codebase they didn't write waste context rediscovering the same facts, answer confidently from stale prose, and have nowhere durable to put what they learn. Teams document systems in wikis that drift silently the day after they're written, because nothing mechanically ties the documentation to the code it describes.

unknown-knowledge solves this by installing a three-store, machine-validated fact architecture into a target repo, plus the agent protocol to use and maintain it:

  • An ontology that maps the system's own concepts — never holding a fact, always pointing at the one source file that owns it, with validators that mechanically catch drift in both directions.
  • A knowledge base for domain/world facts the codebase responds to — citation-anchored, human-gated.
  • A decisions store for the team's rationale — schema'd, lifecycle-tracked, referenceable.
  • An engine of small deterministic validators/extractors/resolvers, and protocol files that teach any coding agent to resolve a request into targeted, current reads instead of repo-wide guessing — and to capture what it learns as it works.

One command (npx unknown-knowledge init) scaffolds everything; one agent-run skill (/kit-bootstrap) crawls the codebase and populates the initial ontology under human gates. After that, the user owns it all.

Distribution & business model (D-015): free and open source — public npm, permissive license (Apache-2.0 recommended) — released to the general community so anyone can stand this up on any project. Ownership-at-init is now backed by license, not just promise, and the extractor library compounds through community contributions as well as the vendor's own engagements. Revenue attaches to services on top (bootstrap engagements, stewardship, custom extractor packs, design-system consulting the kit funnels toward), never to the code.

Goals

  • Any agent, on any supported platform, can take a user request and gather exactly the current facts it needs via one targeted read per fact.
  • Vocabulary/structure drift between the stores and the code is caught mechanically, in both directions, at session time and (when the client wires it) in CI.
  • The system provably improves from use: sessions capture findings; a gated weekly loop turns corroborated findings into approved fixes.
  • Standing this up in a fresh repo takes an afternoon, not an engagement.

Non-goals

  • Behavioral drift detection. The kit catches conceptual/vocabulary drift. Whether code behaves as described is the test suite's job. The kit's docs say this explicitly to prevent over-selling.
  • A maintained platform. No update channel, no telemetry, no client dependency on the vendor post-init (see D-001).
  • Deep knowledge-base authorship. Init delivers shelving + stubs + a handful of cited leaves; depth accrues through the gated promotion loop.

2. Product principles

  1. The map is never the fact. Stores hold claims and pointers; source files hold facts. Agents always follow the pointer and read the source. Prose in any store is navigation, not truth.
  2. Truth anchor determines governance. Every fact is owned by the artifact (ontology), the world (knowledge), or the team (decisions). Litmus test: "if the repo were deleted, would this still be true?" The anchor dictates verifiability, write gate, and staleness policy — which is why the stores cannot be merged.
  3. Deterministic verification, agent-fed. Agents propose, draft, and discover; only boring, versioned, tested code validates. An agent-authored parser never runs in the same session that wrote it — a subtly wrong extractor produces a false all-clear, which is worse than no check.
  4. Seeded once, then owned. The kit is a scaffolding tool, not a platform. Client repos are self-contained forever. The kit improves through the vendor's pilots (miss-logs → new extractor kinds) and reaches future inits only.
  5. Human-gated by default, autonomy earned per category. All store mutations beyond logging start gated. Graduation to autonomy is per change-category, evidence-based, and recorded in the decisions store.
  6. Checkability is refactoring pressure. When a fact can't be extracted because the code never reified it, the right proposal is often a registry/enum in the code — pushing the system toward being describable — not a bespoke parser.

3. The three stores

All stores are YAML, share one navigational grammar (_catalog.yaml → _rules.yaml → class files / tree leaves), and cross-reference each other by ID — never by embedding content across store lines.

3.1 Ontology entry schema

- id: K-210                      # K-NNN, within the class's declared range
  term: Sport
  class: 200-sportsbook          # class files define ranges; taxonomy is per-project
  summary: A bettable sport offered by the sportsbook vertical.
  definition: >                  # prose = navigation, never the fact
    ...
  aliases: [sport type]
  source-of-truth: [src/verticals/sportsbook/sports/registry.ts]   # file OR folder
  owned-by: sportsbook
  used-by: [K-220]               # typed refs; must resolve
  confusable-with: []            # explicit disambiguation, must resolve
  rationale: [D-004]             # optional refs into decisions/
  status: active                 # active | draft | deprecated | proposed
  last-verified: "2026-07-07"
  enumerates:                    # OPTIONAL — presence makes the concept rung-2
    - kind: ts-const-array
      source: src/verticals/sportsbook/sports/registry.ts
      symbol: SUPPORTED_SPORTS
      values: [nfl, nba, mlb, nhl, soccer, tennis]

Folder pointers: source-of-truth may name a directory when the concept is a grouping (a vertical). Rule the bootstrap skill follows: point at a folder for identity, at a file for facts — and pair load-bearing folder pointers with the grouping's entry file so the existence check has teeth.

Rungs. Every concept is born knowing its checkability rung: (1) existence — pointer resolves; (2) value agreement — enumerates re-derived from source and diffed both ways; (3) reverse-audit coverage; (4) prose + last-verified, maintained by the reverification sweep.

3.4 Findings & logs (agent-writable stream, fragment-based)

# logs/findings/2026-07-07-a3f2.yaml — ONE FILE PER ENTRY (D-010)
date: "2026-07-07"
trigger: correction              # correction | recurrence | retrieval-struggle |
                                 # retrieval-miss | quarantine (engine-attributed)
session: <platform/session hint>
summary: User corrected the claimed sport list; concept K-210 was stale.
consulted: { concepts: [K-210], leaves: [] }
status: open                     # open | proposed | resolved | rejected

Fragment-based by design: hundreds of concurrent agent sessions on hundreds of branches append entries; one-file-per-entry means merges never conflict. The reflect skill consolidates fragments into its recommendation list and prunes resolved ones. Sibling logs, same discipline: logs/misses/ (unextractable anchors → extractor backlog) and logs/gaps/ (requests no protocol/skill could route).

3.5 Identity, lifecycle & equality conventions

The boring conventions that cannot be retrofitted under seeded-once — normative for KK-02/05/07 and the steward guide.

  • Schema versioning. Every store file carries schema-version; schema evolution is additive-only (the D-013 rule, generalized).
  • ID policy. IDs are minted at PR time within class ranges, leaving gaps (K-210, K-220…). Published IDs are immutable — never renumbered. Concurrent-mint collision (two branches both create K-210; textually clean merge, duplicate-id error on main): the later-merging PR renumbers its own entry and its own inbound refs — recipe in the steward guide. Knowledge notation is immutable once any reference cites it; moves happen by new leaf + class-elsewhere redirect, never by renaming in place. Decisions are drafted with provisional date-suffixed ids; the steward assigns the final D-NNN at acceptance (anyone may draft a proposed entry through the normal PR gate — that is the decisions store's write path).
  • Ontology status semantics. draft/proposed: structural checks only; resolver downranks; preflight verdict is unknown. active: all checks blocking-grade. deprecated: pointer/value checks demote to warnings; inbound refs allowed with a warning; resolver surfaces it flagged. Retirement recipe: deprecate → strip pointers/enumerates → archive. The source-deletion escape hatch: a PR deleting an anchor deprecates its concept in the same commit — deprecation is precisely the state that stops the blocking path-existence check from dead-ending a legitimate deletion.
  • Enumerates equality. Values are strings, compared byte-exact and case-sensitive, as sets (order irrelevant; duplicates in source are a finding). The loader hard-errors on non-string scalars — YAML 1.1 coercion is a real trap (no → false, on → true): quote such values. Each extractor kind documents which facet it emits (Swift case names vs raw values; an emit: option where both are legitimate).
  • Multi-entry source-of-truth. The first entry is the primary owner; additional entries are secondary references only. Every enumerates descriptor's source must name a listed entry. Two reified declarations of the same value set = two descriptors with identical values (the documented equality idiom); richer cross-concept set relations are deferred (§11.1).

4. The engine

TypeScript/Node, zero build step, minimal dependencies (target: js-yaml and little else). Runs with node directly. Deterministic only — no LLM calls, no network. Everything below is a small module with exit-code semantics so it works as a CI gate, a pre-commit hook, or an agent preflight without adaptation.

ModuleCommandRole
Store loader(library)Parses all three stores once into an indexed in-memory model with diagnostics on one error/warning scale. Every other surface consumes these same diagnostics — the single-health-model guarantee that validator, audit, and preflight can never disagree.
Structural validatorengine/validate.jsBlocking-grade. IDs well-formed & in class range; no duplicates; every source-of-truth path exists; all typed refs (used-by, confusable-with, relates-to, cross-refs) resolve; decision chains acyclic; knowledge tree indexes consistent, no orphans, citations present.
Value validatorengine/validate-values.jsBlocking-grade. Runs every enumerates recipe: parses the named source, extracts the actual value set, diffs against the claim in both directions (value-not-in-source / source-value-missing). Detects the wrong-pointer signature (all claimed values missing from a real file). A malformed descriptor is a hard error, never skipped — an unprovable claim silently passing would re-open the gap this validator closes.
Reverse auditengine/audit.jsAdvisory. Scans code → stores: exported symbols, registries, directories with no matching concept; stale last-verified. Emits typed findings, each carrying a drafted concept YAML proposal ready for human review. Proposal-first by design — never a CI gate.
Resolverengine/resolve.js "query"Scored term matching over terms/aliases/summaries with confusable-with surfaced for disambiguation; returns concepts + their SSOT pointers, plus knowledge catalog entry points. Plain CLI so any agent that can run a shell command gets resolution — no MCP required (deferred, see ledger).
Survey mapengine/survey-map.jsDeterministic traversal-surface builder — the token-efficiency backbone of bootstrap and audit. Git-tracked files only (git ls-files), built-in denylist (dotfolders, vendored/generated dirs, lockfiles, *.gen.*, binaries), per-directory extension/count histograms, and an anchor-candidate pre-scan (regex signatures shared with the extractor kinds). Emits a ranked candidate list + a proposed include/exclude scope. Agents triage this artifact; they never raw-traverse the repo. Honors survey-scope.yaml once it exists.
Preflightengine/preflight.js --concepts <ids>Deterministic verdict module (D-011) — joins the loader's diagnostics to the concepts a task resolved, emitting one verdict per concept (trusted / quarantined / unknown) plus next actions, JSON-first. Store-wide failures degrade all requested verdicts to unknown. Carries the engine exit-code contract: a check that never ran is a blocking defect, never a silent pass. No verdict caching in v1 — a stale "trusted" is a false all-clear. The engine computes verdicts only; conduct on a verdict is protocol-layer policy (§7).

5. Extractor library

Engine contract (uniform across all modules). Exit codes: 0 = clean, 1 = findings/quarantines, 2 = engine or environment failure — a check that never ran is a blocking defect, never a silent pass. Diagnostics carry a closed enum of typed codes on the one health model (parse-error, dangling-pointer, duplicate-id, value-not-in-source, source-value-missing, …) so protocol conduct rules key on codes, not prose. All JSON output is stable and sorted, with no wall-clock timestamps inside diffable output (audit's drafted YAML included — dates injectable), so run-over-run diffs mean something. And the guarantee that doubles as a security answer (D-014): the engine never executes client code — no import/eval/spawn of repo content, parsing is lexical only, no network; a future remote-config adapter must be explicitly opt-in. Pinned by an A6 grep test.

An extractor kind is a small deterministic recipe (~30-line parser) that reads a value set out of a reified anchor. Kinds describe the shape of a declaration, not the file type — .tsx/.jsx are extensions handled by the TS/JS kinds, not separate kinds.

Parsing is regex-level by design; a recipe that can't parse hard-errors loudly instead of hiding. That guarantee only covers detectable failure — so every kind also declares its syntactic envelope and hard-errors when out-of-envelope sentinels appear in the matched span (Swift #if conditional compilation, TS spread/computed keys, re-exports): a confident wrong parse is a false all-clear, the D-005/D-012 failure class. Fixtures plant adversarial-but-extractable shapes, not just unextractable ones (KK-14/15).

5.2 New-kind pipeline (the trust boundary)

1. SURVEY    read-only: the bootstrap/reflect agent inventories anchor shapes
2. MATCH     configure concepts from the shipped kind library
3. DRAFT     for each miss: agent writes the parser
             + a test fixture (sample file → expected values, following
               the shipped extractor-fixture template, D-009)
             + a demo run against the live anchor it was drafted for
4. GATE      human review — fast, because the claim is mechanical
5. INTEGRATE parser lands as versioned, tested code (client zone of their repo;
             kit library if the vendor harvests it for future inits)
6. RE-RUN    concepts depending on the new kind emit last
Hard rule (D-005): validators execute only vendored, versioned, test-covered code — never code authored in the session that runs it. Weigh every bespoke-parser proposal against the alternative of reifying the anchor into a standard shape (principle 6).

6. Init: two phases, two executors

Pattern validated against mattpocock/skills (npx skills add … then /setup-…): the CLI stays dumb and deterministic; all judgment lives in skills executed by whatever agent the client already uses.

Phase 1 — deterministic scaffold (the CLI)

    is deferred (§11.1). Shared instruction files that already exist

  → exits printing: "now run /kit-bootstrap in your agent"

Phase 2 — agent bootstrap (/kit-bootstrap, the client's agent)

  1. Survey (read-only): run engine/survey-map.js first — the agent starts from its ranked anchor-candidate list and directory map, triaging candidates rather than traversing the repo. Raw traversal is a protocol violation; grep is free, reading is expensive, judgment is what the agent is for.
  2. Scope + taxonomy proposal human gate: one combined review — "here's how I read your codebase": the proposed include/exclude scope (which top-level areas are product surface vs. vendored/generated/infra) and the per-project ontology class spine. Reviewing this checklist in the agent conversation is easy; typing it in a terminal would be hell — which is why scoping is Phase-2 work, not an init prompt. Confirmed scope persists as survey-scope.yaml (client zone), reused by every future audit and reflect sweep; a retrieval-miss finding may later propose widening it.
  3. Concept emission: one concept per anchor, born with its rung — enumerates where extractable, folder-identity for verticals, prose + last-verified otherwise. Deliberately not exhaustive: rung-1/2 coverage plus the high-value prose concepts; the reverse audit grows the map proposal-first from there. An ontology born complete is born wrong.
  4. Miss-log: unextractable anchors recorded with shape descriptions → the demand-driven extractor backlog.
  5. Knowledge skeleton: domain elicitation from code vocabulary + structured human interview; emit catalog + divisions + standing-room stubs; a handful of cited leaves human-gated writes.
  6. Green finish: both validators pass; first decisions entries recorded (taxonomy acceptance, any bootstrap trade-offs).
  7. Resumable by construction: on entry the skill detects existing artifacts (scope file, class spine, emitted concepts) and resumes from the first incomplete step; concept emission is idempotent by anchor identity; the confirmed scope+taxonomy gate is never re-litigated on resume — a dead session costs the remaining steps, not a restart.

7. Agent integration & runtime loop

All intelligence lives in platform-agnostic protocol/ markdown: AGENTS.md (navigation contract, runtime loop, gate rules) and the skills (kit-bootstrap, kit-reflect, kb-build, kit-audit). Per-platform wrappers are thin pointers generated at init for every selected platform. Adding a platform later = adding a wrapper.

The runtime loop (every request)

request
  → RESOLVE    engine/resolve.js: request terms → scored concepts
               (confusable-with disambiguation) + knowledge catalog descent
  → PREFLIGHT  engine/preflight.js --concepts <resolved ids> → deterministic
               per-concept verdicts (trusted / quarantined / unknown), JSON.
               CONDUCT on a verdict is protocol policy, client-editable (D-011).
               Recommended default: quarantined concept → continue the task,
               treat the concept as untrusted (gather falls back to
               survey-scoped search), auto-append a quarantine finding.
               Engine failure / check-never-ran → stop (blocking defect).
  → GATHER     follow each source-of-truth pointer and READ THE FILE.
               One targeted read per fact. Prose and enumerates are navigation.
  → ACT        execute. Before committing, run resolve.js --paths <changed
               files> (reverse lookup over the loader's pointer index) to
               list every concept the diff touches; update those concepts
               IN THE SAME COMMIT.
  → RECORD     append Findings on any of the five triggers — each Finding's
               consulted: refs ARE the consultation trail (no separate
               trail store in v1); log gaps and misses.

  RESOLVE returns nothing (common in month one — bootstrap is deliberately
  non-exhaustive): fall back to search WITHIN survey-scope.yaml, and append
  a retrieval-miss finding only when the topic plausibly should be mapped
  (an unmapped area the scope excludes is expected, not a miss).
  preflight.js with an empty --concepts list validates store health only
  and exits on the store verdict.
Why PREFLIGHT matters without CI: the kit is CI-agnostic (D-006) — validators are plain commands the client wires into CI whenever they're ready. Until then, PREFLIGHT makes every agent session a validation gate. CI hardens the gate from "every session" to "every commit"; nothing blocks on it.

8. Self-improvement loop (v1: capture → consolidate → gate)

Capture always-on, autonomous appends

The protocol obligates the agent — any session, any platform — to append a Finding when a trigger fires:

TriggerSignal
correctionuser states the agent / a store is wrong
recurrencethe same thing is asked for repeatedly
retrieval-strugglefound, but slowly or indirectly (resolver missed; aliases lacking)
retrieval-misscould not find it at all
quarantineengine-attributed: preflight flagged a concept as untrusted and the session proceeded in degraded mode (auto-appended; volume per concept is the steward's priority signal)

Structured YAML appends; cheap; no judgment at capture time.

Capture content policy (KK-20): summaries reference concept IDs and file paths — never verbatim user text, quoted session content, or secrets; the session hint is an opaque ID. Committed fragments are permanent git history in the client's repo and are reviewable content like any other PR change.

Consolidate — /kit-reflect human-run, daily/weekly

Reads the findings log, clusters recurring signals, emits a recommendation list: each item a concrete diff (fix a concept, add an alias, repoint an SSOT, promote a finding to knowledge, draft an extractor) with its justifying findings attached as evidence. Human approves/rejects per item; approved diffs apply; the loop closes by re-running the relevant validator (pass → resolved + verified date; fail → re-opened). Rejections record the reason. Recurrences re-open the same entry, never duplicate. Three hygiene rules keep the queue trustworthy at scale: open findings still uncorroborated after N reflect cycles are archived with a rollup note (N tunable — unbounded open fragments otherwise drown reflect by month ten); clusters containing mutually contradictory corrections are flagged disputed and resolved by reading the SSOT — a dispute never counts as corroboration; and reflect stamps a last-reflect state file and records per-item approval outcome by category (approved / approved-with-modification / rejected) — the observable basis for the trust-graduation trigger, which is otherwise unmeasurable post-handoff.

Evidence standard: any gated change requires multiple corroborating findings — one correction is a data point, three are a pattern. Single-occurrence noise never reaches the review queue, or the humans stop trusting it and the graduation path dies before it starts.

Trust graduation designed-for, not built in v1

Autonomy is per change-category, never global: e.g. "alias additions" may graduate to autonomous after N consecutive approved-without-modification cycles, while "SSOT repoints" stay gated forever. Each graduation is recorded as a decisions/ entry of category trust — the third store governs changes to the system's own trust boundary. v1 ships the schema and the documented path; no autonomous mutations.

Operating at team scale (100s of engineers)

The kit has no runtime — everything is files in the repo — so git is the concurrency model (D-010): hundreds of agent sessions ride hundreds of branches, isolated by git, merged by PRs, gated by validators. No server, no locks, no shared state, no new approval tooling.

  • The ordinary engineer invokes nothing. The protocol rides along in every session: their agent resolves/preflights/gathers during normal work (they just notice it's faster and righter), updates touched concepts in the same commit (the store change rides their normal PR), and appends findings silently when triggers fire. The daily UX is invisibility with receipts.
  • The steward role. A small group or rotation owns the loop: runs /kit-reflect on cadence (weekly; daily at high finding volume), triages reverse-audit proposals, gates kb-build promotions. Recommended mechanism: a CODEOWNERS rule scoped to ontology/ knowledge/ decisions/ protocol/ engine/ — logs/ explicitly excluded: findings ride nearly every agent-assisted PR, and gating them would drown stewards in rubber-stamping or teach engineers to delete fragments to unblock merges. The human gates run on review infrastructure the org already trusts. Reflect output lands as a PR, so even the improvement loop's own changes pass normal review and CI. Honest caveat, stated in the steward guide: findings travel only with merged PRs — abandoned branches lose theirs; corroboration tolerates a lossy denominator.
  • The loop has a heartbeat. kit-audit reports days-since-last-reflect (from the last-reflect stamp), open-fragment counts, and top-N quarantined concepts — a lapsed steward rotation is visible instead of silent. The deferred dashboards trigger (§11.1) can only fire if someone can see the queue; this is the minimal instrument that makes it observable.
  • Scale strengthens the evidence standard. Corroboration across many engineers' sessions is far stronger signal than repetition from one — the multi-corroboration threshold gets more trustworthy, not noisier, as the team grows.
  • Logs are fragment-based (one file per entry, §3.4) precisely so concurrent appends never merge-conflict; reflect consolidates and prunes.
  • CI stops being optional in practice. D-006 stands — init never wires CI — but the seeded docs state plainly: session-level PREFLIGHT is a sufficient gate for a small team, not for hundreds of engineers. At team scale, wire the two validator commands into CI.

11.3 Assumptions

13. Seed decisions (the kit's own decisions store, day one)

IDDecisionRationale (condensed)
D-001Seeded-once-then-owned; no update channel, everThe kit is a scaffolding tool, not a platform. User independence is the positioning promise — now license-backed (D-015); improvements reach future inits via the vendor's engagements and community contributions.
D-002Engine is TypeScript/Node, zero build step, minimal depsMost target shops are web/platform-agnostic TS; Node is ubiquitous in CI; clients must be able to read and extend what they own — no binaries.

D-004Two-phase init: deterministic CLI scaffold, then agent-run bootstrapThe CLI stays dumb; all judgment lives in skills run by the client's own agent. Validated pattern (mattpocock/skills).
D-005Validators run only vendored, versioned, tested code — never same-session agent-authored codeA subtly wrong parser yields a false all-clear; determinism is the product. Agents draft; humans gate; code validates.
D-006CI-agnostic; init never touches CI configRemoves an adoption blocker; PREFLIGHT makes every agent session a gate meanwhile; docs cover wiring when the client is ready.
D-007Init copies from an explicit payload allowlist; acceptance fixtures and kit tests can never shipNothing leaks by omission; the manifest is the boundary.
D-008Behavioral drift detection is out of scope, stated openlyThe kit catches conceptual/vocabulary drift; behavior is the test suite's job. Saying so prevents over-selling and misplaced trust.

D-014The engine never executes client code: no import/eval/spawn of repo content, lexical parsing only, no network; any future adapter that breaks this (e.g. remote-config fetch-at-validate) must be explicitly opt-inBoth a security answer ("does this tool run our code in CI?" — "no, guaranteed and grep-tested," A6) and a design constraint that stops the tempting "just import the module and read the array" extractor from silently crossing the line. Apache-2.0's patent grant eases the security reviews this tool will face, and its trademark clause protects the product name. 14. Glossary (domain model)

Ships as CONTEXT.md in the kit repo. Terms resolved during the grilling session, 2026-07-07.

Kit
The white-label product: a CLI that scaffolds the three stores, engine, and agent protocol into a target repo. Seeded-once-then-owned.
Engine
The vendored deterministic code: loader, validators, extractors, resolver. Never an agent; agents feed it and read it.
Store
One of three governed YAML fact repositories, distinguished by truth anchor: Ontology (artifact), Knowledge (world), Decisions (team).
Truth anchor
Who owns a fact's truth. Determines store membership and all downstream governance. Litmus: "repo deleted — still true?"
Anchor (extraction)
A reified structure declaring a fact machine-readably: registry, enum, config, directory convention.
Extractor kind
A small deterministic recipe reading a value set from an anchor. Agent-drafted, human-gated, deployed only as versioned tested code.
Rung
A fact's checkability level: (1) existence, (2) value agreement, (3) reverse-audit coverage, (4) dated prose. Every concept is born knowing its rung.
Enumerates block
A concept's claimed value set plus the recipe to re-derive it from source. Drift = the diff, both directions.
Runtime loop
resolve → preflight → gather → act → record. The per-request agent protocol.
Finding
A structured note appended when a trigger fires (correction, recurrence, retrieval-struggle, retrieval-miss, quarantine — the fifth is engine-attributed). Raw signal; judged only at reflect time; content policy: concept IDs and paths, never verbatim user text. Uncorroborated entries age out to archived after N reflect cycles.
Reflect skill
Human-run consolidation clustering findings into a per-item gated recommendation list. Multi-corroboration required for gated changes.
Trust graduation
Future per-category autonomy upgrades, each recorded as a Decisions entry. Designed-for in v1, not built.
Miss-log
Record of anchors no extractor kind can read — the demand-driven backlog for new kinds.
Payload manifest
The explicit allowlist of what init copies. Nothing ships by omission.
Survey map
The deterministic traversal-surface artifact (engine/survey-map.js): git-tracked files minus a built-in denylist, directory histograms, and a regex anchor-candidate pre-scan. Agents triage it; they never raw-traverse the repo.
Survey scope
survey-scope.yaml — the human-confirmed include/exclude boundary from bootstrap gate 1. Reused by every audit and reflect sweep; widened via findings, never re-litigated from scratch.
Steward
The person or rotation owning the improvement loop in a client repo: runs reflect on cadence, triages audit proposals, gates knowledge promotions. Wired in via a CODEOWNERS rule on the kit root; approves reflect output as ordinary PRs (D-010).
Verdict
Preflight's per-concept output: trusted / quarantined / unknown. Computed deterministically by the engine; what an agent does about a verdict is protocol-layer policy the client owns (D-011).
Suppression
A client-zone record (term, sourcePath, reason, date) telling the reverse audit "this is deliberately not a concept." Exact-match only in v1 — can only under-suppress, never falsely silence (D-013). Rejection memory for the steward.
Stack
A language/config ecosystem the extractor library covers (Swift, TS/JS, config). Selected at init (auto-detected, confirmable); drives which extractor fixtures ship. Distinct from agent platform, the IDE/agent axis that drives wrapper generation.
Acceptance fixture
A synthetic codebase (Swift, TS) in the kit repo with known anchors, planted drift, and unextractable shapes. Acceptance target; never in the payload (D-007).
Extractor fixture
A per-kind sample-file → expected-values pair. Ships to client repos for their selected stacks as the runnable extractor gate and the authoring template for client-drafted kinds (D-009). Later stacks are client-authored — warned at init.
Protocol files
Platform-agnostic markdown holding all agent intelligence; per-IDE wrappers are thin pointers to them.
